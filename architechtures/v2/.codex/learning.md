- This container initially had `torch 2.4.1+cu124`, which cannot execute on the local Blackwell `sm_120` GPU (`RTX PRO 4500 Blackwell`); local autoresearch runs required upgrading to `torch 2.9.1+cu130` before CUDA training would work.
- For the current 83M next-token JEPA setup, lighter auxiliary weights plus `weight_decay=0` beat the previous ledger best: `future_token_83m_auxlighter.yaml` (`lambda_jepa=0.2`, `beta_sigreg=0.005`, `ema=0.99`, warmups `1000`) reached `val_bpb=3.485299` at 5k steps before CUDA compile was made mandatory.
- Pushing auxiliaries even lighter than that (`lambda_jepa=0.15`, `beta_sigreg=0.003`) improved the 1k anchor BPB (`3.483177`) but produced much worse JEPA/SIGReg losses and unstable latent-scale growth by the 2.5k eval of the 5k continuation; treat that branch as a discard unless the objective explicitly prioritizes very short-horizon BPB over representation health.
- The LM loader already provides shifted next-token labels (`x=t[0..L-1]`, `y=t[1..L]`), so `next_token_loss` must score `CE(logits, labels)` directly. A second internal shift trains `t -> t+2` and can silently destroy BPB while still looking numerically healthy.
- The same no-second-shift rule applies to `autoresearch/train.py:evaluate_bpb()`: BPB must score `outputs["logits"]` directly against the loader-shifted `labels`. The old wrapper sliced `logits[:, :-1]` against `labels[:, 1:]`, effectively evaluating `t -> t+2` and badly inflating BPB.
- The top JEPA block should use the same-depth EMA teacher as the lower JEPA blocks. The old special frozen output-target encoder was a contract-breaking exception and is now legacy-only for checkpoint compatibility.
- The frozen `output_target_*` modules and custom legacy checkpoint-key migration were removed; the model now exposes a single JEPA target contract and does not support pre-cleanup checkpoint layouts.
- A fully local auxiliary-gradient variant that detached each block input before JEPA/SIGReg encoder computation regressed badly (`val_bpb=4.249308` at 5k on the 83M auxlighter wd0 baseline). Treat strict layer-local aux backprop as a discard for this repo unless the objective changes away from BPB.
- Residual-output projections now use scaled init `0.02 / sqrt(2 * depth)` instead of the generic `0.02`. This applies to attention `out_proj`, JEPA `projector`, and the final MLP output projection, while the other projections keep the standard init.
- JEPA and SIGReg auxiliary losses should operate on layer-stacked tensors (`G, B, L, K`) where possible; keep the total objective as the sum of per-layer means while avoiding per-layer Python loss loops and mask-based tensor shrinking.
- EMA JEPA targets are CE-only: reuse the student's cached `h_l_post_attn` and apply EMA `ce_norm + compressor`; do not run attention in the teacher target path.
- The current production-size config is the 44.24M-trainable CE-only EMA model (`D=768`, `K=384`, `depth=10`, tied embeddings) with aux-light settings `lambda_jepa=0.2`, `beta_sigreg=0.005`, warmups `1000`, and EMA `0.99`; this is ~42.2 MiB student-only int8 or ~46.0 MiB including EMA.
- Current best compact config name: `compact_16m_auxlight.yaml`. Current best compact run name: `compact16m-5k-rowwise-auxlight` with `val_bpb=2.376269` at 5k.
- For the 16M compact model, moving from `lambda_jepa=1.0, beta_sigreg=0.1, ema=0.996` to aux-light settings (`ema=0.99`, `lambda_jepa=0.2`, `beta_sigreg=0.005`, warmups `1000`) removed the catastrophic final-state/logit collapse and improved `val_bpb` from `3.489307` to `2.376269` at 5k.
- Short 5k->5.25k continuation probes on that improved compact checkpoint showed essentially identical LM loss for `lambda_jepa=0.2` vs `0.15`, but lower total loss and slightly lower SIGReg loss for the lighter branch. Prefer `compact_16m` with `lambda_jepa=0.15` and longer warmups `2000` for the next compact sweep.
- A fresh from-scratch 5k rerun invalidated that continuation-based preference: `compact16m-5k-rowwise-auxlighter2k` (`lambda_jepa=0.15`, warmups `2000`) regressed versus `compact16m-5k-rowwise-auxlight` at every eval point, finishing at `val_bpb=2.388355` vs `2.376269` and slightly worse `eval_loss_lm`. For compact sweeps, keep the stronger aux-light settings (`lambda_jepa=0.2`, warmups `1000`) as the default until an architectural change lands.
- The weaker compact YAML increased latent anisotropy without improving LM: on direct checkpoint probes it had the same top-1 accuracy (`19.58%`) but worse NLL (`3.9468` vs `3.9054`), lower mean top-1 confidence (`0.1796` vs `0.1912`), higher final/logit top-PC fraction, and notably worse `z` concentration in layers `2`, `5`, and `8`.
- Resume from checkpoints loaded with `map_location=device` requires restoring both CPU and CUDA RNG states as CPU `torch.uint8` tensors before calling `torch.set_rng_state` / `torch.cuda.set_rng_state_all`; otherwise resume crashes with `TypeError: RNG state must be a torch.ByteTensor`.
- A direct `projector(z_l + delta_l)` writeback couples the JEPA latent path too tightly to the residual transition. A 1k compact probe with a separate JEPA-block residual `transition_mlp(h_l_post_attn)` improved `eval/loss_lm` at step 999 (`4.6088` vs `4.7552` for the original compact run at the same horizon), but absolute BPB was still poor (`2.6831`) and the new checkpoint layout is not backward-compatible with old `projector` checkpoints without explicit migration.
- RoPE is now the positional contract: token embeddings are token-only, every attention layer applies cached RoPE to split-head `q/k` using precomputed fp32 `cos/sin` buffers up to `max_length`, and legacy checkpoints may carry an unused `embeddings.position_embedding.weight` key that should be ignored on weight load. Optimizer-state resume is not supported for those legacy pre-RoPE checkpoints.
- Attention now routes through `utils.flash_attention`: it tries `flash_attn.cute.flash_attn_func` on CUDA tensors with zero dropout and otherwise falls back to PyTorch SDPA. On the local RTX PRO 4500 Blackwell environment, the installed `flash-attn-4` import required `setuptools`, and a direct runtime smoke still failed inside the CUTE kernel stack, so the wrapper must be allowed to disable flash at runtime and continue with SDPA.
- JEPA/SIGReg layer selection is controlled by `auxiliary_layer_start` plus `auxiliary_layer_stride`: skip the first `start` JEPA blocks, then supervise every `stride` blocks. For the compact 16M top-third probe, use `auxiliary_layer_start=6`, `auxiliary_layer_stride=1`, and `jepa_dropout_rate=0.1`.
- Autoresearch can override JEPA dropout/start/stride from the CLI; when those flags are used, it writes a run-local `autoresearch_config.yaml` and passes that resolved config path to the real trainer so the wrapper stays thin.
- Multi-horizon JEPA schedules live in `auxiliary_target_groups`; each group names explicit JEPA layer indices, `target_type` (`same_layer` or `final_layer`), inclusive future horizon range, and optional weight. When groups are present they replace the older start/stride selection, and omitted layers receive no JEPA/SIGReg auxiliary loss.

- Compact multi-horizon grouped JEPA with six active layers regressed versus top-third: `compact16m-5k-horizon-groups-t2-20260425` used layers 3-4 same-layer `t+2..5`, 5-6 same-layer `t+2..9`, and 7-8 final-layer `t+2..12` with 10% aux dropout, finishing at `val_bpb=2.126356` versus `2.114118` for top-third + 10% dropout. The path ran correctly, but broad six-layer horizon supervision appears too heavy at the 5k-step compact budget.

- Direct top-third state prediction to final-layer future span regressed: `compact16m-5k-topthird-finalspan-state-20260425` used layers 6-8, `prediction=state`, `target_type=final_layer`, horizon `t+2..9`, and 10% aux dropout. It finished at `val_bpb=2.130347` versus `2.114118` for top-third delta prediction. JEPA loss was lower (`1.937029`) but SIGReg was much higher (`44.171875`), so direct state matching is not a current default at unchanged `beta_sigreg`.
- The current best compact run is `compact16m-5k-topthird-mean-t1-t4-20260425`: layers 6-8, same-layer mean future target `t+1..t+4`, 10% aux dropout, `lambda_jepa=0.2`, `beta_sigreg=0.005`, `val_bpb=2.111987`. Heavier weights on the same setup (`lambda_jepa=0.25`, `beta_sigreg=0.0075`) regressed to `val_bpb=2.120223`.
- Final-layer future teacher without SIGReg is a discard for now: `compact16m-5k-topthird-final-future-nosigreg-t1-t4-20260425` used layers 6-8 predicting final-layer future `t+1..t+4` with `beta_sigreg=0.0` and finished at `val_bpb=2.143707`, much worse than the same-layer mean target despite low JEPA loss.
- Split content+dynamics latents with a same-total 50/50 split are a discard for the current compact setup and were rolled back from active code: `compact16m-5k-topthird-mean-t1-t4-split-latents-20260425` used `content_dim=112`, `dynamics_dim=112` for old `compressed_dim=224` and regressed to `val_bpb=2.206675` versus `2.111987` current best. SIGReg fell (`4.382812`) but LM loss worsened (`3.790255`), and the dynamics branch became scale-unstable with active-layer `z_variance` reaching hundreds/thousands.
- Zero-gated JEPA residual adapters are implemented and tensor-contract tested, but the current compact config is a discard for BPB: `compact16m-5k-topthird-mean-t1-t4-gated-adapter-20260425` used layers 6-8 with normalized `z + delta` residual writeback and finished at `val_bpb=2.124057` versus `2.111987` current best. The gates learned only tiny values near zero, so the model mostly ignored the adapter.
- Future-span JEPA masks must apply sequence `valid_mask` to both current and future positions. A deterministic audit caught that the forward path only masked current positions, so padded future tokens could enter auxiliary targets; `jepa_future_valid_mask` now also intersects `sequence_valid_mask`, and tests reconstruct the strict future mask manually.
- GQA is config-gated through `num_kv_heads`: omitted values preserve full MHA, while compact GQA probes use `num_heads=8`, `num_kv_heads=4`. Attention now stores split `c_q/c_k/c_v` projections; legacy packed `c_attn.weight` checkpoints migrate into split weights when the target config has matching full-MHA K/V shapes.
- Clean same-code GQA comparison on 2026-04-25: current-code full MHA control `compact16m-5k-topthird-mean-t1-t4-currentcode-20260425` reached `val_bpb=2.114893`, while `num_heads=8,num_kv_heads=4` GQA reached `val_bpb=2.128756`. GQA removed 2,007,040 trainable attention parameters (`21,525,504 -> 19,518,464` trainable total), lowered peak memory (`0.556 -> 0.521` GB), but barely changed wall time (`234.7 -> 231.2` s) at this short sequence/batch. Do not compare the GQA row directly against older commit `04ed391` as a same-code ablation.
