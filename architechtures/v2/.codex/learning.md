- `torch.compile` on the v1 trainer should not see per-step Python ints inside `IntertwinedHJEPA.forward`; passing `step` into warmup logic caused Dynamo to specialize on `step == N` and recompile every iteration. Compute JEPA/SIGReg warmup scalars in the trainer and pass them as tensor inputs instead.
- SIGReg randomness must stay graph-safe for compile. `Tensor.item()`, per-step Python seeding, and `torch.Generator(...)` construction inside `forward()` cause Dynamo graph breaks and restart storms. Plain `torch.randn(...)` inside the graph compiles cleanly.
- CUDA bf16 autocast with fp32 `nn.RMSNorm` weights emits a mixed-dtype warning and falls off the fused path. A small `RMSNorm(nn.RMSNorm)` subclass that computes in fp32 and casts back removes the warning while keeping state_dict/test compatibility.
- Import TorchAO lazily in the trainer so normal non-TorchAO runs do not emit TorchAO extension-version warnings at startup.
- In the JEPA target path, target source and target basis are separate stability levers. On the 125.5M real-data run, `EMA_CE_l(x_{l+1})` behaved much better at 5k than `EMA_CE_{l+1}(x_{l+1})`, so switching to the next layer's CE basis appears to add its own instability on top of the original post-attention target issue.
- On top of the better target path, the projector input also matters: keeping `z` in the update value but detaching it (`projector(z.detach() + delta)`) beat both the plain `z + delta` path and `delta`-only at 5k, which points to the live `z` gradient through the projector as another instability source.
- When tightening the EMA target path to match JEPA’s “student learns, teacher is EMA” rule, the most compact implementation here is one frozen EMA block copy per JEPA block plus a narrow `encode_context()` method. That keeps the target path full-width enough without exploding the bookkeeping into separate EMA submodule lists.
- Persist per-step metrics directly from `scripts/train_intertwined_hjepa.py` into `run_dir/metrics.jsonl`. Console logs and the final TSV row are not enough to reconstruct loss curves later.
- Latent-space losses now normalize at the source. `jepa_delta_loss` compares RMS-normalized `delta`, `z`, and `target_z`, and SIGReg consumes RMS-normalized compressed states. Keep cross-entropy unchanged on logits; only representation-space losses should be normalized.
- Future-token JEPA works best as a token-axis shift on teacher latents while keeping tensor shapes fixed: build the teacher latent at shape `(B, L, K)`, shift it left by one token into the target tensor, and enforce validity with an explicit `(B, L)` JEPA mask that drops the tail positions instead of shortening tensors.
- For the current future-token JEPA experiment, lower JEPA layers should use the same layer's EMA encoder on `states[layer_index]` as the raw teacher basis before the token shift. Reusing the next layer's EMA basis mixes temporal prediction with a second basis change and makes the ablation harder to interpret.
